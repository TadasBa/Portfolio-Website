Vilnius University gave me a better sense of how software projects fail, not only how they are built. The useful lessons were often practical: write smaller changes, document decisions, test what matters, and keep the system understandable.

## First year: learning how to study

The first year was mostly about adjusting. Programming assignments, group work, and theory-heavy courses forced me to become more consistent. I learned that writing code is only one part of the work. Reading requirements, asking better questions, and explaining decisions mattered just as much.

## Project work

University projects made the trade-offs more visible. A feature could work locally and still be hard to maintain. A design could look fine and still be awkward to extend. Those projects helped me care more about structure, naming, and clear boundaries.

## Thesis work

My bachelor’s thesis compared LLM-based code quality assessment with static analysis methods for C# code. It was useful because it connected research, tooling, and software engineering practice. It also made the limits of automated assessment clearer.

## What carried into work

The most useful habits were simple: break work into smaller parts, keep interfaces clear, write tests around risky behavior, and avoid complexity that does not solve a real problem.

That is still how I try to build software.
